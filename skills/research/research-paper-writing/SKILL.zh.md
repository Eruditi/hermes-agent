# 研究论文写作流程

端到端的研究论文写作流程，专为**NeurIPS、ICML、ICLR、ACL、AAAI和COLM**等顶级会议的ML/AI论文而设计。本技能覆盖完整的研究生命周期：实验设计、执行、监控、分析、论文写作、评审、修订和提交。

这**不是线性流程** — 而是一个迭代循环。结果会触发新的实验。评审会触发新的分析。代理必须处理这些反馈循环。

```
┌─────────────────────────────────────────────────────────────┐
│                    研究论文写作流程                          │
│                                                             │
│  阶段0: 项目设置 ──► 阶段1: 文献综述                          │
│       │                          │                          │
│       ▼                          ▼                          │
│  阶段2: 实验设计     阶段5: 论文起草 ◄──┐                  │
│       │                          │                   │      │
│       ▼                    阶段6: 自我评审      │      │
│  阶段3: 执行与监控           & 修订 ──────────┘      │
│       │                          │                          │
│       ▼                    阶段7: 提交                      │
│  阶段4: 分析 ─────► (反馈到阶段2或5)                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 何时使用此技能

在以下情况使用此技能：
- **开始新的研究论文**，基于现有代码库或想法
- **设计和运行实验**以支持论文观点
- **撰写或修订**研究论文的任何部分
- **准备提交**到特定会议或研讨会
- **回应评审意见**，进行额外实验或修订
- **在不同会议格式之间转换**论文
- **撰写非实证论文** — 理论、综述、基准测试或立场论文（见[超越实证ML的论文类型](#超越实证ml的论文类型)）
- **设计人类评估**用于NLP、HCI或对齐研究
- **准备录用后的交付物** — 海报、演讲、代码发布

## 核心理念

1. **主动出击**。提供完整的草稿，而不是问题。科学家很忙 — 产生他们可以反应的具体内容，然后迭代。
2. **永远不要编造引用**。AI生成的引用有~40%的错误率。始终通过编程获取。将无法验证的引用标记为`[需要引用]`。
3. **论文是一个故事，不是实验的集合**。每篇论文都需要一个明确的贡献，用一句话表达。如果你做不到这一点，论文就还没准备好。
4. **实验服务于观点**。每个实验必须明确说明它支持哪个观点。永远不要运行与论文叙述无关的实验。
5. **尽早提交，经常提交**。每个完成的实验批次，每个论文草稿更新 — 用描述性消息提交。Git日志是实验历史。

### 主动性与协作

**默认：主动出击。先起草，带着草稿提问。**

| 自信程度 | 行动 |
|---------|------|
| **高**（代码库清晰，贡献明确） | 撰写完整草稿，交付，根据反馈迭代 |
| **中**（有些模糊） | 撰写带有标记不确定性的草稿，继续进行 |
| **低**（主要未知） | 通过`clarify`提出1-2个有针对性的问题，然后起草 |

| 部分 | 自主起草？ | 随草稿标记 |
|------|------------|------------|
| 摘要 | 是 | "将贡献表述为X — 如需调整" |
| 引言 | 是 | "强调了问题Y — 如不正确请纠正" |
| 方法 | 是 | "包含了细节A、B、C — 添加缺失部分" |
| 实验 | 是 | "突出了结果1、2、3 — 如需重新排序" |
| 相关工作 | 是 | "引用了论文X、Y、Z — 添加我遗漏的任何内容" |

**仅在以下情况阻塞输入**：目标会议不明确，多种相互矛盾的框架，结果似乎不完整，明确要求先评审。

## 阶段0：项目设置

**目标**：建立工作空间，理解现有工作，确定贡献。

### 步骤0.1：探索代码库

```bash
# 了解项目结构
ls -la
find . -name "*.py" | head -30
find . -name "*.md" -o -name "*.txt" | xargs grep -l -i "result\|conclusion\|finding"
```

寻找：
- `README.md` — 项目概述和观点
- `results/`、`outputs/`、`experiments/` — 现有发现
- `configs/` — 实验设置
- `.bib`文件 — 现有引用
- 草稿文档或笔记

### 步骤0.2：组织工作空间

建立一致的工作空间结构：

```
workspace/
  paper/               # LaTeX源文件、图表、编译后的PDF
  experiments/         # 实验运行脚本
  code/                # 核心方法实现
  results/             # 原始实验结果（自动生成）
  tasks/               # 任务/基准定义
  human_eval/          # 人类评估材料（如需）
```

### 步骤0.3：设置版本控制

```bash
git init  # 如果尚未初始化
git remote add origin <repo-url>
git checkout -b paper-draft  # 或main
```

**Git纪律**：每个完成的实验批次都用描述性消息提交。例如：
```
Add Monte Carlo constrained results (5 runs, Sonnet 4.6, policy memo task)
Add Haiku baseline comparison: autoreason vs refinement baselines at cheap model tier
```

### 步骤0.4：确定贡献

在撰写任何内容之前，明确：
- **是什么**：这篇论文贡献的单一事物是什么？
- **为什么**：有什么证据支持它？
- **那又怎样**：读者为什么要关心？

> 向科学家提议："根据我的理解，主要贡献是：[一句话]。关键结果显示[Y]。这是你想要的框架吗？"

### 步骤0.5：创建待办事项列表

使用`todo`工具创建结构化项目计划：

```
研究论文待办事项：
- [ ] 定义一句话贡献
- [ ] 文献综述（相关工作 + 基线）
- [ ] 设计核心实验
- [ ] 运行实验
- [ ] 分析结果
- [ ] 撰写初稿
- [ ] 自我评审（模拟评审者）
- [ ] 根据评审进行修订
- [ ] 提交准备
```

在整个项目中更新此列表。它作为跨会话的持久状态。

### 步骤0.6：估算计算预算

在运行实验之前，估算总成本和时间：

```
计算预算清单：
- [ ] API成本：（模型每token价格）×（每次运行估计token数）×（运行次数）
- [ ] GPU小时：（每次实验时间）×（实验数量）×（种子数量）
- [ ] 人类评估成本：（标注者）×（小时）×（小时费率）
- [ ] 总预算上限和应急资金（为重新运行增加30-50%）
```

在实验运行时跟踪实际支出：
```python
# 简单的成本跟踪模式
import json, os
from datetime import datetime

COST_LOG = "results/cost_log.jsonl"

def log_cost(experiment: str, model: str, input_tokens: int, output_tokens: int, cost_usd: float):
    entry = {
        "timestamp": datetime.now().isoformat(),
        "experiment": experiment,
        "model": model,
        "input_tokens": input_tokens,
        "output_tokens": output_tokens,
        "cost_usd": cost_usd,
    }
    with open(COST_LOG, "a") as f:
        f.write(json.dumps(entry) + "\n")
```

**当预算紧张时**：在承诺全面扫描之前，运行试点实验（1-2个种子，任务子集）。使用更便宜的模型调试管道，然后切换到目标模型进行最终运行。

### 步骤0.7：多作者协调

大多数论文有3-10位作者。尽早建立工作流程：

| 工作流程 | 工具 | 何时使用 |
|---------|------|----------|
| **Overleaf** | 基于浏览器 | 多位作者同时编辑，无git经验 |
| **Git + LaTeX** | 带`.gitignore`的`git`用于aux文件 | 技术团队，需要基于分支的评审 |
| **Overleaf + Git同步** | Overleaf高级版 | 两者的最佳组合 — 实时协作与版本历史 |

**部分所有权**：为每个部分分配一位主要作者。其他人评论但不直接编辑。防止合并冲突和风格不一致。

```
作者协调清单：
- [ ] 就部分所有权达成一致（谁写什么）
- [ ] 设置共享工作空间（Overleaf或git仓库）
- [ ] 建立符号约定（在任何人写作之前）
- [ ] 安排内部评审轮次（不仅仅在最后）
- [ ] 指定一人进行最终格式检查
- [ ] 在创建图表之前就图表风格（颜色、字体、大小）达成一致
```

**尽早达成一致的LaTeX约定**：
- `\method{}`宏用于一致的方法命名
- 引用风格：`\citet{}` vs `\citep{}`的使用
- 数学符号：向量用小写粗体，矩阵用大写粗体等
- 英式 vs 美式拼写

## 阶段1：文献综述

**目标**：找到相关工作，确定基线，收集引用。

### 步骤1.1：识别种子论文

从代码库中已引用的论文开始：

```bash
# 通过终端：
grep -r "arxiv\|doi\|cite" --include="*.md" --include="*.bib" --include="*.py"
find . -name "*.bib"
```

### 步骤1.2：搜索相关工作

**加载`arxiv`技能**进行结构化论文发现：`skill_view("arxiv")`。它提供arXiv REST API搜索、Semantic Scholar引用图、作者简介和BibTeX生成。

使用`web_search`进行广泛发现，`web_extract`获取特定论文：

```
# 通过web_search：
web_search("[主要技术] + [应用领域] site:arxiv.org")
web_search("[基线方法] comparison ICML NeurIPS 2024")

# 通过web_extract（用于特定论文）：
web_extract("https://arxiv.org/abs/2303.17651")
```

尝试的其他搜索查询：

```
搜索查询：
- "[主要技术] + [应用领域]"
- "[基线方法] comparison"
- "[问题名称] state-of-the-art"
- 现有引用中的作者姓名
```

**推荐**：安装**Exa MCP**进行实时学术搜索：
```bash
claude mcp add exa -- npx -y mcp-remote "https://mcp.exa.ai/mcp"
```

### 步骤1.2b：深化搜索（广度优先，然后深度）

平面搜索（一轮查询）通常会错过重要的相关工作。使用受深度研究管道启发的迭代**广度然后深度**模式：

```
迭代文献搜索：

第1轮（广度）：4-6个平行查询，涵盖不同角度
  - "[方法] + [领域]"
  - "[问题名称] state-of-the-art 2024 2025"
  - "[基线方法] comparison"
  - "[替代方法] vs [你的方法]"
  → 收集论文，提取关键概念和术语

第2轮（深度）：根据第1轮学习生成后续查询
  - 第1轮论文中发现的新术语
  - 最相关的第1轮结果引用的论文
  - 需要调查的矛盾发现
  → 收集论文，识别剩余差距

第3轮（针对性）：填补特定差距
  - 第1-2轮中识别的缺失基线
  - 并发工作（过去6个月，相同问题）
  - 关键负面结果或失败方法
  → 当新查询返回的大多是你已经见过的论文时停止
```

**何时停止**：如果一轮返回>80%的论文已经在你的集合中，搜索就饱和了。通常2-3轮就足够了。对于综述论文，预计需要4-5轮。

**对于基于代理的工作流程**：通过`delegate_task`并行委托每轮查询。收集结果，去重，然后从组合学习中生成下一轮查询。

### 步骤1.3：验证每个引用

**永远不要从记忆中生成BibTeX。始终通过编程获取。**

对于每个引用，遵循强制性的5步流程：

```
引用验证（每个引用必须执行）：
1. 搜索 → 使用特定关键字查询Semantic Scholar或Exa MCP
2. 验证 → 确认论文存在于2+个来源（Semantic Scholar + arXiv/CrossRef）
3. 检索 → 通过DOI内容协商获取BibTeX（通过编程，而非记忆）
4. 验证 → 确认你引用的观点确实出现在论文中
5. 添加 → 将验证的BibTeX添加到参考文献
如果任何步骤失败 → 标记为[需要引用]，通知科学家
```

```python
# 通过DOI获取BibTeX
import requests

def doi_to_bibtex(doi: str) -> str:
    response = requests.get(
        f"https://doi.org/{doi}",
        headers={"Accept": "application/x-bibtex"}
    )
    response.raise_for_status()
    return response.text
```

如果你无法验证引用：

```latex
\cite{PLACEHOLDER_author2024_verify_this}  % TODO: 验证此引用是否存在
```

**始终告诉科学家**："我已将[X]个引用标记为需要验证的占位符。"

请参阅[references/citation-workflow.md](references/citation-workflow.md)获取完整的API文档和完整的`CitationManager`类。

### 步骤1.4：组织相关工作

按方法组织论文，而非逐个论文：

**好**："一类工作使用X的假设[引用]，而我们使用Y的假设因为..."
**坏**："Smith等人介绍了X。Jones等人介绍了Y。我们结合了两者。"

## 阶段2：实验设计

**目标**：设计直接支持论文观点的实验。每个实验必须回答特定问题。

### 步骤2.1：将观点映射到实验

创建明确的映射：

| 观点 | 实验 | 预期证据 |
|------|------|----------|
| "我们的方法优于基线" | 主要比较（表1） | 胜率，统计显著性 |
| "对于较弱的模型效果更大" | 模型缩放研究 | 单调改进曲线 |
| "收敛需要范围约束" | 约束vs非约束 | 收敛率比较 |

**规则**：如果一个实验不映射到观点，就不要运行它。

### 步骤2.2：设计基线

强大的基线是区分被接受论文和被拒绝论文的关键。评审者会问："他们与X比较了吗？"

标准基线类别：
- **朴素基线**：最简单的可能方法
- **强大基线**：最佳已知现有方法
- **消融基线**：你的方法减去一个组件
- **计算匹配基线**：相同的计算预算，不同的分配

### 步骤2.3：定义评估协议

在运行任何内容之前，指定：
- **指标**：你要测量的内容，方向符号（越高/越低越好）
- **聚合**：如何跨运行/任务组合结果
- **统计测试**：什么测试将建立显著性
- **样本大小**：多少次运行/问题/任务

### 步骤2.4：编写实验脚本

遵循成功研究管道的这些模式：

**增量保存** — 每步后保存结果以进行崩溃恢复：
```python
# 每个问题/任务后保存
result_path = f"results/{task}/{strategy}/result.json"
if os.path.exists(result_path):
    continue  # 跳过已完成的工作
# ... 运行实验 ...
with open(result_path, 'w') as f:
    json.dump(result, f, indent=2)
```

**工件保存** — 保存所有中间输出：
```
results/<experiment>/
  <task>/
    <strategy>/
      final_output.md          # 最终结果
      history.json             # 完整轨迹
      pass_01/                 # 每次迭代的工件
        version_a.md
        version_b.md
        critic.md
```

**关注点分离** — 将生成、评估和可视化分开：
```
run_experiment.py              # 核心实验运行器
run_baselines.py               # 基线比较
run_comparison_judge.py        # 盲评估
analyze_results.py             # 统计分析
make_charts.py                 # 可视化
```

请参阅[references/experiment-patterns.md](references/experiment-patterns.md)获取完整的设计模式、cron监控和错误恢复。

### 步骤2.5：设计人类评估（如适用）

许多NLP、HCI和对齐论文需要人类评估作为主要或补充证据。在运行自动化实验之前设计这一点 — 人类评估通常有更长的准备时间（IRB批准，标注者招募）。

**何时需要人类评估**：
- 自动化指标无法捕获你关心的内容（流畅性、有用性、安全性）
- 你的贡献与人类面对的品质有关（可读性、偏好、信任）
- NLP会议（ACL、EMNLP）的评审者期望生成任务的人类评估

**关键设计决策**：

| 决策 | 选项 | 指导 |
|------|------|------|
| **标注者类型** | 专家、众包工作者、最终用户 | 与你的观点要求匹配 |
| **规模** | Likert（1-5）、成对比较、排名 | 成对比较比Likert对LLM输出更可靠 |
| **样本大小** | 每个标注者和总项目 | 功效分析或至少100个项目，3+个标注者 |
| **一致性指标** | Cohen's kappa、Krippendorff's alpha、ICC | Krippendorff's alpha用于>2个标注者；也报告原始一致性 |
| **平台** | Prolific、MTurk、内部团队 | Prolific用于质量；MTurk用于规模；内部用于领域专业知识 |

**标注指南清单**：
```
- [ ] 清晰的任务描述和示例（好的和坏的）
- [ ] 模糊情况的决策标准
- [ ] 每个类别至少2个工作示例
- [ ] 注意力检查/黄金标准项目（总数的10-15%）
- [ ] 资格任务或筛选轮次
- [ ] 每个项目的估计时间和公平补偿（≥当地最低工资）
- [ ] 如机构要求，进行IRB/伦理审查
```

**报告要求**（评审者会检查所有这些）：
- 标注者数量及其资格
- 带有特定指标和值的标注者间一致性
- 补偿细节（金额，估计小时费率）
- 标注界面描述或截图（附录）
- 总标注时间

请参阅[references/human-evaluation.md](references/human-evaluation.md)获取完整指南，包括人类评估数据的统计测试、众包质量控制模式和IRB指导。

## 阶段3：实验执行与监控

**目标**：可靠地运行实验，监控进度，从失败中恢复。

### 步骤3.1：启动实验

对长时间运行的实验使用`nohup`：

```bash
nohup python run_experiment.py --config config.yaml > logs/experiment_01.log 2>&1 &
echo $!  # 记录PID
```

**并行执行**：同时运行独立实验，但要注意API速率限制。同一API上4+个并发实验会减慢每个实验的速度。

### 步骤3.2：设置监控（Cron模式）

对于长时间运行的实验，设置定期状态检查。cron提示应遵循此模板：

```
监控提示模板：
1. 检查进程是否仍在运行：ps aux | grep <pattern>
2. 读取日志的最后30行：tail -30 <logfile>
3. 检查已完成的结果：ls <result_dir>
4. 如果存在结果，读取并报告：cat <result_file>
5. 如果全部完成，提交：git add -A && git commit -m "<描述性消息>" && git push
6. 以结构化格式报告（带有关键指标的表格）
7. 回答此实验的关键分析问题
```

**静默模式**：如果自上次检查以来没有变化，用``响应以抑制对用户的通知。只在有消息时报告。

### 步骤3.3：处理失败

常见失败模式和恢复：

| 失败 | 检测 | 恢复 |
|------|------|------|
| API速率限制/信用耗尽 | 日志中的402/429错误 | 等待，然后重新运行（脚本跳过已完成的工作） |
| 进程崩溃 | PID消失，结果不完整 | 从最后一个检查点重新运行 |
| 困难问题超时 | 进程卡住，日志无进展 | 杀死并跳过，在结果中注明 |
| 错误的模型ID | 引用模型名称的错误 | 修复ID并重新运行 |

**关键**：脚本应始终检查现有结果并跳过已完成的工作。这使重新运行安全且高效。

### 步骤3.4：提交已完成的结果

每个实验批次完成后：

```bash
git add -A
git commit -m "Add <experiment name>: <key finding in 1 line>"
git push
```

### 步骤3.5：维护实验日志

Git提交跟踪发生了什么，但不是**探索树** — 基于你所学内容决定下一步尝试什么的决策。维护一个结构化的实验日志，捕获此树：

```json
// experiment_journal.jsonl — 每个实验尝试追加一个条目
{
  "id": "exp_003",
  "parent": "exp_001",
  "timestamp": "2025-05-10T14:30:00Z",
  "hypothesis": "添加范围约束将修复exp_001的收敛失败",
  "plan": "使用max_tokens=2000和固定结构模板重新运行autoreason",
  "config": {"model": "haiku", "strategy": "autoreason", "max_tokens": 2000},
  "status": "completed",
  "result_path": "results/exp_003/",
  "key_metrics": {"win_rate": 0.85, "convergence_rounds": 3},
  "analysis": "范围约束修复了收敛。胜率从0.42跃升至0.85。",
  "next_steps": ["在Sonnet上尝试相同的约束", "测试无结构模板"],
  "figures": ["figures/exp003_convergence.pdf"]
}
```

**为什么需要日志，而不仅仅是git？** Git跟踪文件更改。日志跟踪推理：你为什么尝试X，你学到了什么，以及这对下一个实验意味着什么。在撰写论文时，这棵树对于方法部分（"我们观察到X，这激发了Y"）和诚实的失败报告非常宝贵。

**选择最佳路径**：当日志显示分支树（exp_001 → exp_002a, exp_002b, exp_003）时，确定最能支持论文观点的路径。在附录中记录死胡同分支作为消融或负面结果。

**每个实验的代码快照**：每次运行后复制实验脚本：
```bash
cp experiment.py results/exp_003/experiment_snapshot.py
```
这即使在后续代码更改后也能实现精确的重现。

## 阶段4：结果分析

**目标**：提取发现，计算统计数据，识别故事。

### 步骤4.1：聚合结果

编写分析脚本：
1. 加载批次中的所有结果文件
2. 计算每个任务和聚合指标
3. 生成摘要表格

```python
# 标准分析模式
import json, os
from pathlib import Path

results = {}
for result_file in Path("results/").rglob("result.json"):
    data = json.loads(result_file.read_text())
    strategy = result_file.parent.name
    task = result_file.parent.parent.name
    results.setdefault(strategy, {})[task] = data

# 计算聚合指标
for strategy, tasks in results.items():
    scores = [t["score"] for t in tasks.values()]
    print(f"{strategy}: mean={np.mean(scores):.1f}, std={np.std(scores):.1f}")
```

### 步骤4.2：统计显著性

始终计算：
- **误差条**：标准偏差或标准误差，指定哪一个
- **置信区间**：关键结果的95% CI
- **成对测试**：用于比较两种方法的McNemar测试
- **效应大小**：Cohen's d或h用于实际显著性

请参阅[references/experiment-patterns.md](references/experiment-patterns.md)获取McNemar测试、引导CI和Cohen's h的完整实现。

### 步骤4.3：识别故事

分析后，明确回答：
1. **主要发现是什么？** 用一句话陈述。
2. **什么让你惊讶？** 意外结果往往成就最好的论文。
3. **什么失败了？** 失败的实验可能是最有信息量的。诚实报告失败会加强论文。
4. **需要哪些后续实验？** 结果往往引发新问题。

#### 处理负面或无效结果

当你的假设错误或结果不确定时，你有三个选择：

| 情况 | 行动 | 会议适合度 |
|------|------|------------|
| 假设错误但**为什么**有信息量 | 将论文框架围绕分析为什么 | NeurIPS、ICML（如果分析严谨） |
| 方法不优于基线但**揭示了新东西** | 将贡献重新框架为理解/分析 | ICLR（重视理解）、研讨会论文 |
| 对流行观点的干净负面结果 | 写出来 — 该领域需要知道 | NeurIPS Datasets & Benchmarks、TMLR、研讨会 |
| 结果不确定，没有明确故事 | 转向 — 运行不同的实验或重新框架 | 不要强迫不存在的论文 |

**如何撰写负面结果论文**：
- 以社区相信的内容和为什么测试它很重要开头
- 描述你的严谨方法（必须无懈可击 — 评审者会更仔细审查）
- 用统计证据清楚地呈现无效结果
- 分析**为什么**预期结果没有实现
- 讨论对该领域的影响

**明确欢迎负面结果的会议**：NeurIPS（Datasets & Benchmarks track）、TMLR、ML Reproducibility Challenge、主要会议的研讨会。一些研讨会专门呼吁负面结果。

### 步骤4.4：创建图表和表格

**图表**：
- 所有图表使用矢量图形（PDF）：`plt.savefig('fig.pdf')`
- 色盲安全调色板（Okabe-Ito或Paul Tol）
- 自包含标题 — 读者应无需正文即可理解
- 图表内无标题 — 标题起此作用

**表格**：
- 使用`booktabs` LaTeX包
- 每个指标的最佳值加粗
- 包括方向符号（越高/越低越好）
- 一致的小数精度

```latex
\usepackage{booktabs}
\begin{tabular}{lcc}
\toprule
Method & Accuracy $\uparrow$ & Latency $\downarrow$ \\
\midrule
Baseline & 85.2 & 45ms \\
\textbf{Ours} & \textbf{92.1} & 38ms \\
\bottomrule
\end{tabular}
```

### 步骤4.5：决定：更多实验还是写作？

| 情况 | 行动 |
|------|------|
| 核心观点得到支持，结果显著 | 进入阶段5（写作） |
| 结果不确定，需要更多数据 | 返回阶段2（设计） |
| 意外发现建议新方向 | 返回阶段2（设计） |
| 缺少评审者会要求的一个消融 | 运行它，然后进入阶段5 |
| 所有实验完成但有些失败 | 记录失败，进入阶段5 |

### 步骤4.6：编写实验日志（连接到写作）

在进入论文写作之前，创建一个结构化的实验日志，连接结果到 prose。这是实验和写作之间最重要的连接组织 — 没有它，写作代理必须从原始结果文件重新推导故事。

**创建`experiment_log.md`**，具有以下结构：

```markdown
# 实验日志

## 贡献（一句话）
[论文的主要观点]

## 运行的实验

### 实验1：[名称]
- **测试的观点**：[这支持哪个论文观点]
- **设置**：[模型、数据集、配置、运行次数]
- **关键结果**：[一句话，包含数字]
- **结果文件**：results/exp1/final_info.json
- **生成的图表**：figures/exp1_comparison.pdf
- **意外发现**：[任何意外的内容]

### 实验2：[名称]
...

## 图表
| 文件名 | 描述 | 属于哪个部分 |
|--------|------|------------|
| figures/main_comparison.pdf | 比较所有方法在基准X上的条形图 | 结果，图2 |
| figures/ablation.pdf | 移除组件A、B、C的消融 | 结果，图3 |
...

## 失败的实验（为诚实而记录）
- [尝试了什么，为什么失败，它告诉我们什么]

## 开放问题
- [结果提出的论文应解决的任何问题]
```

**为什么这很重要**：在起草时，代理（或委派的子代理）可以加载`experiment_log.md`和LaTeX模板，生成基于实际结果的初稿。没有这个桥梁，写作代理必须解析原始JSON/CSV文件并推断故事 — 这是产生幻觉或错误报告数字的常见原因。

**Git纪律**：将此日志与它描述的结果一起提交。

## 迭代改进：策略选择

此管道中的任何输出 — 论文草稿、实验脚本、分析 — 都可以迭代改进。autoreason研究提供了经验证据，说明每种改进策略何时有效，何时失败。使用本节选择正确的方法。

### 快速决策表

| 你的情况 | 策略 | 为什么 |
|---------|------|--------|
| 中层模型 + 约束任务 | **Autoreason** | 最佳点。生成-评估差距最宽。基线积极破坏弱模型输出。 |
| 中层模型 + 开放任务 | **带范围约束的Autoreason** | 添加固定事实、结构或可交付成果以限制改进空间。 |
| 前沿模型 + 约束任务 | **Autoreason** | 即使在前沿也赢得2/3的约束任务。 |
| 前沿模型 + 开放任务 | **批评和修订**或**单次通过** | Autoreason最后。模型自我评估足够好。 |
| 具体技术任务（系统设计） | **批评和修订** | 直接查找和修复循环更高效。 |
| 模板填充任务（一个正确结构） | **单次通过**或**保守** | 最小决策空间。迭代无价值。 |
| 带测试用例的代码 | **Autoreason（代码变体）** | 结构化分析*为什么*失败后再修复。恢复率62% vs 43%。 |
| 非常弱的模型（Llama 8B类） | **单次通过** | 模型太弱，无法生成多样化候选。投资于生成质量。 |

### 生成-评估差距

**核心洞察**：Autoreason的价值取决于模型生成能力和自我评估能力之间的差距。

```
模型层级        │ 生成能力 │ 自我评估 │ 差距    │ Autoreason价值
────────────────┼──────────┼──────────┼────────┼─────────────────
弱（Llama 8B）   │ 差       │ 差       │ 小     │ 无 — 无法生成多样化候选
中（Haiku 3.5）  │  decent  │ 差       │ 大     │ 最大 — 42/42完美Borda
中（Gemini Flash）│ decent  │ 中等     │ 大     │ 高 — 赢得2/3
强（Sonnet 4）   │ 好       │  decent  │ 中等   │ 中等 — 赢得3/5
前沿（S4.6）     │ 优秀     │ 好       │ 小     │ 仅带约束
```

这个差距是结构性的，不是暂时的。随着成本下降，今天的前沿成为明天的中层。最佳点移动但永远不会消失。

### Autoreason循环（摘要）

每次通过从新鲜、隔离的代理生成三个候选：

1. **批评者** → 发现现有A的问题（无修复）
2. **作者B** → 基于批评修订A
3. **综合者** → 合并A和B（随机标签）
4. **评审小组** → 3个盲CoT评审者通过Borda计数对A、B、AB进行排名
5. **收敛** → A连续赢k=2次通过 → 完成

**关键参数**：
- k=2收敛（k=1过早，k=3太昂贵，无质量增益）
- 始终使用CoT评审者（3倍更快收敛）
- 温度0.8作者，0.3评审者
- 保守平局：现有者赢得平局
- 每个角色都是无共享上下文的新鲜代理

### 应用于论文草稿

当通过autoreason改进论文本身时：
- **向批评者提供 ground truth**：实际实验数据、结果JSON、统计输出。没有这些，模型会幻觉编造的消融研究和虚假置信区间。
- **至少使用3个工作评审者**：损坏的评审者解析器不会添加噪声 — 它会完全阻止平衡。
- **范围约束修订**："解决这些特定弱点"而不是"改进论文"。

### 失败模式

| 失败 | 检测 | 修复 |
|------|------|------|
| 无收敛（A从未赢） | A在20+次通过中赢<15% | 为任务添加范围约束 |
| 综合漂移 | 字数无限增长 | 约束结构和可交付成果 |
| 低于单次通过的退化 | 基线得分高于迭代输出 | 切换到单次通过；模型可能太弱 |
| 过度拟合（代码） | 公共测试通过高，私有测试通过低 | 使用结构化分析，不仅仅是测试反馈 |
| 损坏的评审者 | 解析失败将小组减少到3以下 | 在继续之前修复解析器 |

请参阅[references/autoreason-methodology.md](references/autoreason-methodology.md)获取完整的提示、Borda评分细节、模型选择指南、范围约束设计模式和计算预算参考。

## 阶段5：论文起草

**目标**：撰写完整的、可发表的论文。

### 大型项目的上下文管理

具有50+实验文件、多个结果目录和广泛文献笔记的论文项目很容易超出代理的上下文窗口。主动管理：

**每个起草任务加载到上下文的内容**：

| 起草任务 | 加载到上下文 | 不加载 |
|---------|------------|--------|
| 撰写引言 | `experiment_log.md`、贡献声明、5-10个最相关的论文摘要 | 原始结果JSON、完整实验脚本、所有文献笔记 |
| 撰写方法 | 实验配置、伪代码、架构描述 | 原始日志、其他实验的结果 |
| 撰写结果 | `experiment_log.md`、结果摘要表格、图表列表 | 完整分析脚本、中间数据 |
| 撰写相关工作 | 组织的引用笔记（步骤1.4输出）、.bib文件 | 实验文件、原始PDF |
| 修订通过 | 完整论文草稿、特定评审者关注点 | 其他一切 |

**原则**：
- **`experiment_log.md`是主要的上下文桥梁** — 它总结了写作所需的一切，而无需加载原始数据文件（见步骤4.6）
- **一次加载一个部分的上下文**。起草方法的子代理不需要文献综述笔记。
- **总结，不要包含原始文件**。对于200行结果JSON，加载10行摘要表格。对于50页相关论文，加载5句摘要 + 你关于其相关性的2行笔记。
- **对于非常大的项目**：创建`context/`目录，包含预压缩摘要：
  ```
  context/
    contribution.md          # 1句话
    experiment_summary.md    # 关键结果表格（来自experiment_log.md）
    literature_map.md        # 组织的引用笔记
    figure_inventory.md      # 带描述的图表列表
  ```

### 叙述原则

**最关键的洞察**：你的论文不是实验的集合 — 它是一个故事，有一个由证据支持的明确贡献。

每篇成功的ML论文都以Neel Nanda所谓的"叙述"为中心：一个简短、严谨、基于证据的技术故事，带有读者关心的结论。

**三大支柱（在引言结束时必须清晰）**：

| 支柱 | 描述 | 测试 |
|------|------|------|
| **是什么** | 1-3个具体的新颖观点 | 你能用一句话陈述它们吗？ |
| **为什么** | 严谨的经验证据 | 实验是否区分你的假设与替代方案？ |
| **那又怎样** | 读者为什么要关心 | 这是否与公认的社区问题相关？ |

**如果你不能用一句话陈述你的贡献，你还没有论文。**

### 时间分配

在以下各项上花费大约**相等的时间**：
1. 摘要
2. 引言
3. 图表
4. 其他所有内容的总和

**为什么？** 大多数评审者在到达你的方法之前就形成判断。读者遇到你的论文的顺序是：标题 → 摘要 → 引言 → 图表 → 可能的其余部分。

### 写作工作流程

```
论文写作清单：
- [ ] 步骤1：定义一句话贡献
- [ ] 步骤2：起草图1（核心思想或最引人注目的结果）
- [ ] 步骤3：起草摘要（5句公式）
- [ ] 步骤4：起草引言（最多1-1.5页）
- [ ] 步骤5：起草方法
- [ ] 步骤6：起草实验和结果
- [ ] 步骤7：起草相关工作
- [ ] 步骤8：起草结论和讨论
- [ ] 步骤9：起草局限性（所有会议要求）
- [ ] 步骤10：计划附录（证明、额外实验、细节）
- [ ] 步骤11：完成论文清单
- [ ] 步骤12：最终评审
```

### 两遍改进模式

当使用AI代理起草时，使用**两遍**方法（在SakanaAI的AI-Scientist管道中被证明有效）：

**第1遍 — 按部分写作 + 立即改进**：
对于每个部分，撰写完整草稿，然后在相同上下文中立即改进。这会在部分新鲜时捕获本地问题（清晰度、流程、完整性）。

**第2遍 — 全论文上下文的全局改进**：
所有部分起草后，在了解完整论文的情况下重新访问每个部分。这会捕获跨部分问题：冗余、不一致的术语、叙述流程以及一个部分承诺而另一部分未交付的差距。

```
第二遍改进提示（每部分）：
"在完整论文的上下文中审查[部分]。
- 它与论文的其余部分是否一致？与其他部分是否有冗余？
- 术语与引言和方法是否一致？
- 有没有什么可以削减而不削弱信息？
- 叙述是否从前一部分流畅地过渡到下一部分？
进行最小、有针对性的编辑。不要从头重写。"
```

### 步骤5.0：标题

标题是论文最常被阅读的元素。它决定是否有人点击进入摘要。

**好标题**：
- 陈述贡献或发现："Autoreason: When Iterative LLM Refinement Works and Why It Fails"
- 突出令人惊讶的结果："Scaling Data-Constrained Language Models"（暗示你可以）
- 命名方法 + 它做什么："DPO: Direct Preference Optimization of Language Models"

**坏标题**：
- 太通用："An Approach to Improving Language Model Outputs"
- 太长：超过~15个单词的任何内容
- 仅行话："Asymptotic Convergence of Iterative Stochastic Policy Refinement"（这是给谁的？）

**规则**：
- 如果你有方法名称，包含它（为了可引用性）
- 包含1-2个评审者会搜索的关键词
- 避免冒号，除非两半都有意义
- 测试：评审者是否仅从标题就知道领域和贡献？

### 步骤5.1：摘要（5句公式）

来自Sebastian Farquhar（DeepMind）：

```
1. 你实现了什么："We introduce...", "We prove...", "We demonstrate..."
2. 为什么这很难且重要
3. 你如何做（使用专业关键词以提高可发现性）
4. 你有什么证据
5. 你最显著的数字/结果
```

**删除**通用开头，如"Large language models have achieved remarkable success..."

### 步骤5.2：图1

图1是大多数读者看的第二件事（在摘要之后）。在撰写引言之前起草它 — 它迫使你澄清核心思想。

| 图1类型 | 何时使用 | 示例 |
|---------|----------|------|
| **方法图** | 新架构或管道 | 显示你的系统的TikZ流程图 |
| **结果预告** | 一个引人注目的结果讲述整个故事 | 条形图："我们的方法 vs 基线"，差距明显 |
| **问题说明** | 问题不直观 | 显示你修复的失败模式的前后对比 |
| **概念图** | 抽象贡献需要视觉基础 | 方法属性的2x2矩阵 |

**规则**：图1必须无需阅读任何文本即可理解。标题 alone 应传达核心思想。有目的地使用颜色 — 不要只是装饰。

### 步骤5.3：引言（最多1-1.5页）

必须包含：
- 清晰的问题陈述
- 简要方法概述
- 2-4个项目符号贡献列表（两列格式，每个最多1-2行）
- 方法应在第2-3页开始

### 步骤5.4：方法

启用重新实现：
- 概念概述或伪代码
- 列出所有超参数
- 架构细节足以重现
- 呈现最终设计决策；消融在实验中

### 步骤5.5：实验与结果

对于每个实验，明确说明：
- **它支持什么观点**
- 它如何与主要贡献联系
- 观察什么："蓝线显示X，这证明了Y"

要求：
- 带有方法的误差条（标准偏差 vs 标准误差）
- 超参数搜索范围
- 计算基础设施（GPU类型，总小时）
- 种子设置方法

### 步骤5.6：相关工作

按方法组织，而非逐个论文。慷慨引用 — 评审者可能是相关论文的作者。

### 步骤5.7：局限性（必需）

所有主要会议都要求这一点。诚实有帮助：
- 评审者被指示不要惩罚诚实的局限性承认
- 通过首先识别弱点来预先阻止批评
- 解释为什么局限性不会破坏核心观点

### 步骤5.8：结论与讨论

**结论**（必需，0.5-1页）：
- 用一句话重述贡献（与摘要不同的措辞）
- 总结关键发现（2-3句，不是列表）
- 影响：这对该领域意味着什么？
- 未来工作：2-3个具体的下一步（不是模糊的"我们将X留给未来工作"）

**讨论**（可选，有时与结论结合）：
- 超出直接结果的更广泛影响
- 与其他子领域的联系
- 对方法何时有效、何时无效的诚实评估
- 实际部署考虑

**不要**在结论中引入新结果或观点。

### 步骤5.9：附录策略

所有主要会议的附录都是无限的，对于可重现性至关重要。结构：

| 附录部分 | 内容 |
|---------|------|
| **证明与推导** | 太长而无法在正文中包含的完整证明。正文可以陈述定理，"证明见附录A"。 |
| **额外实验** | 消融、缩放曲线、每个数据集的分解、超参数敏感性 |
| **实现细节** | 完整的超参数表、训练细节、硬件规格、随机种子 |
| **数据集文档** | 数据收集过程、标注指南、许可、预处理 |
| **提示与模板** | 使用的确切提示（对于基于LLM的方法）、评估模板 |
| **人类评估** | 标注界面截图、给标注者的说明、IRB细节 |
| **额外图表** | 每个任务的分解、轨迹可视化、失败案例示例 |

**规则**：
- 主要论文必须自包含 — 评审者不需要阅读附录
- 永远不要只在附录中放置关键证据
- 交叉引用："完整结果见表5（附录B）"，而不仅仅是"见附录"
- 使用`\appendix`命令，然后是`\section{A: Proofs}`等

### 页面预算管理

当超过页面限制时：

| 削减策略 | 节省 | 风险 |
|---------|------|------|
| 将证明移至附录 | 0.5-2页 | 低 — 标准做法 |
| 浓缩相关工作 | 0.5-1页 | 中 — 可能错过关键引用 |
| 将表格与子图组合 | 0.25-0.5页 | 低 — 通常提高可读性 |
| 谨慎使用`\vspace{-Xpt}` | 0.1-0.3页 | 低如果微妙，高如果明显 |
| 移除定性示例 | 0.5-1页 | 中 — 评审者喜欢示例 |
| 减小图表大小 | 0.25-0.5页 | 高 — 图表必须保持可读 |

**不要**：减小字体大小，更改边距，移除必需部分（局限性，更广泛影响），或对正文使用`\small`/`\footnotesize`。

### 步骤5.10：伦理与更广泛影响声明

大多数会议现在要求或强烈鼓励伦理/更广泛影响声明。这不是样板 — 评审者阅读它，并且可以标记触发桌面拒绝的伦理问题。

**应包括的内容**：

| 组件 | 内容 | 由谁要求 |
|------|------|----------|
| **积极的社会影响** | 你的工作如何造福社会 | NeurIPS、ICML |
| **潜在的负面影响** | 滥用风险、双重使用担忧、失败模式 | NeurIPS、ICML |
| **公平与偏见** | 你的方法/数据是否有已知偏见？ | 所有会议（隐含） |
| **环境影响** | 大规模训练的计算碳足迹 | ICML，越来越多的NeurIPS |
| **隐私** | 你的工作是否使用或启用个人数据处理？ | ACL、NeurIPS |
| **LLM披露** | AI是否用于写作或实验？ | ICLR（强制性）、ACL |

**撰写声明**：

```latex
\section*{Broader Impact Statement}
% NeurIPS/ICML: 在结论之后，不计入页面限制

% 1. 积极应用（1-2句）
This work enables [specific application] which may benefit [specific group].

% 2. 风险和缓解（1-3句，具体）
[Method/model] could potentially be misused for [specific risk]. We mitigate
this by [specific mitigation, e.g., releasing only model weights above size X,
including safety filters, documenting failure modes].

% 3. 影响声明的局限性（1句）
Our evaluation is limited to [specific domain]; broader deployment would
require [specific additional work].
```

**常见错误**：
- 写"我们预见没有负面影响"（几乎永远不真实 — 评审者不信任这一点）
- 模糊："这可能被滥用"而不具体说明如何
- 忽略大规模工作的计算成本
- 忘记在要求的会议上披露LLM使用

**计算碳足迹**（对于训练繁重的论文）：
```python
# 使用ML CO2 Impact工具方法估算
gpu_hours = 1000  # 总GPU小时
gpu_tdp_watts = 400  # 例如，A100 = 400W
pue = 1.1  # 电源使用效率（数据中心开销）
carbon_intensity = 0.429  # kg CO2/kWh（美国平均；因地区而异）

energy_kwh = (gpu_hours * gpu_tdp_watts * pue) / 1000
carbon_kg = energy_kwh * carbon_intensity
print(f"Energy: {energy_kwh:.0f} kWh, Carbon: {carbon_kg:.0f} kg CO2eq")
```

### 步骤5.11：数据表与模型卡（如适用）

如果你的论文引入**新数据集**或**发布模型**，请包含结构化文档。评审者越来越期望这一点，NeurIPS Datasets & Benchmarks track要求这一点。

**数据集的数据表**（Gebru等人，2021）— 包含在附录中：

```
数据集文档（附录）：
- 动机：为什么创建此数据集？它支持什么任务？
- 组成：实例是什么？有多少？什么数据类型？
- 收集：数据如何收集？来源是什么？
- 预处理：应用了什么清理/过滤？
- 分发：数据集如何分发？在什么许可下？
- 维护：谁维护它？如何报告问题？
- 伦理考虑：包含个人数据？获得同意？
  潜在危害？已知偏见？
```

**模型卡**（Mitchell等人，2019）— 模型发布的附录中包含：

```
模型卡（附录）：
- 模型详细信息：架构、训练数据、训练过程
- 预期用途：主要用例，超出范围的用途
- 指标：评估指标和基准测试结果
- 伦理考虑：已知偏见，公平性评估
- 局限性：已知失败模式，模型表现不佳的领域
```

### 写作风格

**句子级清晰度（Gopen & Swan的7原则）**：

| 原则 | 规则 |
|------|------|
| 主谓接近 | 保持主语和动词接近 |
| 强调位置 | 在句子末尾放置强调 |
| 主题位置 | 先放上下文，后放新信息 |
| 旧信息先于新信息 | 熟悉信息 → 不熟悉信息 |
| 一个单元，一个功能 | 每个段落提出一个观点 |
| 行动在动词中 | 使用动词，而非名词化 |
| 上下文先于新信息 | 在呈现之前设置舞台 |
